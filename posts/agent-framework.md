# Agent 框架的构建与实践

自主 AI Agent 正在改变我们构建软件的方式。本文分享我在实际项目中的框架选型、架构设计与优化经验。

## 为什么需要 Agent 框架

传统 LLM 调用是「一问一答」，而 Agent 需要：

- **工具调用** — 连接搜索、数据库、API 等外部能力
- **多步推理** — 规划 → 执行 → 观察 → 修正的循环
- **状态管理** — 维护对话上下文与任务进度
- **可观测性** — 记录每一步决策，便于调试

## 核心架构模式

### ReAct 模式

Reasoning + Acting 是最常用的范式：模型先思考下一步，再选择工具执行，根据结果继续推理。

```
Thought → Action → Observation → Thought → ...
```

### 工具设计原则

1. **单一职责** — 每个工具只做一件事
2. **清晰描述** — Prompt 中的 tool description 决定调用准确率
3. **错误友好** — 返回结构化错误，让 Agent 能自我修正

## 实践建议

- 从 2–3 个核心工具开始，逐步扩展
- 为关键路径编写 eval 测试集
- 限制最大迭代次数，避免无限循环
- 记录 token 消耗与延迟，持续优化 Prompt

## 总结

Agent 框架的价值在于把 LLM 的「智能」与工程化的「可控」结合起来。选对框架、设计好工具边界，是落地成功的关键。
